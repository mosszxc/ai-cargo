**Дата:** 2026-03-19 (updated 2026-03-20)

## Фаза 0: Spike (дни 1–2)

**Цель:** убить главный риск ДО того как писать код.

Главный риск — парсинг 1688. Если не сможем надёжно вытаскивать данные — MVP мёртв.

### Задачи

- [ ] Собрать 20 разных ссылок на 1688 (одежда, электроника, обувь, мебель, игрушки)
- [ ] Playwright headless → попробовать извлечь: название, цену, вес, габариты
- [ ] Зафиксировать: на скольких из 20 удалось? Какие поля пустые?
- [ ] Проверить антибот: есть ли капча, блокировка по IP
- [ ] Попробовать через прокси + ротация user-agent
- [ ] Написать `docs/spike-results.md` с результатами

### Go/No-go

| Результат | Решение |
|---|---|
| >70% ссылок парсятся | Идём дальше |
| 50–70% | Идём, но с fallback на ручной ввод |
| <50% | Ищем альтернативу (API 1688, сторонний сервис) |

## Правила процесса

1. **Spike первый** — ни строчки продакшн-кода, пока не убедились что 1688 парсится
2. **Вертикально** — сначала весь путь «ссылка → расчёт» end-to-end, потом статусы
3. **Тесты на каждой фазе** — критерий готовности ДО перехода к следующей фазе
4. **Реальные данные с фазы 2** — не мокаем, используем реальные ссылки и ставки
5. **Один клиент перед десятью** — онбордим одну компанию, вылизываем, потом масштабируем
6. **Логи всегда** — если что-то сломалось и нет лога, это наша ошибка

### Фаза 5: Пилот — DONE когда:

- [ ] Менеджер компании умеет обновлять статусы (обучён)
- [ ] Собрана обратная связь: что работает, что нет, чего не хватает
- [ ] Метрики за неделю зафиксированы: кол-во расчётов, точность, время ответа
- [ ] Критические баги пофикшены
