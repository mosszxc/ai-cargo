# PRD: AI-Ассистент для авто-карго Китай→Россия

**Версия:** 1.0
**Дата:** 2026-03-19
**Статус:** MVP

---

## 1. Продукт

AI-ассистент для карго-компаний на маршруте Китай→Россия. Работает в Telegram через OpenClaw. Три функции:

1. **Мгновенный расчёт** — клиент кидает ссылку на товар с 1688 + количество → получает полный расчёт за 30 сек по ставкам конкретной компании
2. **Автостатусы фур** — менеджер пишет «фура 025 на границе» → 20–50 клиентам этой фуры автоматически улетает уведомление
3. **Голосовые сообщения** — менеджер и клиент могут отправлять голосовые на русском и китайском → Whisper транскрибирует → OpenClaw обрабатывает как текст

Естественный язык + голос, без команд. Менеджер общается с ботом как с человеком — текстом или голосом.

## 2. Пользователи

**Структура ботов:**
- Один бот на компанию — менеджер и клиенты используют ОДИН и тот же бот
- Бот определяет роль по Telegram ID: если ID совпадает с `manager_telegram_id` компании — это менеджер, все остальные — клиенты
- **@AiCargoManager_bot** — тестовый бот MVP. Менеджер (Telegram ID 5093456686) имеет полный доступ
- **@CargoSystemProd_bot** — зарезервирован, в MVP не используется

### Менеджер/владелец карго-компании
- Обновляет статусы фур: «фура 025 прошла границу» (текст или голосовое)
- Управляет ставками: «обнови авто до 2.90»
- Привязывает клиентов к фурам: «добавь @ivan к фуре 025»
- Видит активные фуры: «покажи все фуры»
- Может делать всё голосом — наговорил в Telegram, бот понял и выполнил

### Клиент карго-компании (конечный пользователь)
- Кидает ссылку на 1688 + количество → получает расчёт
- Пишет свободным текстом: «500 кг одежда Гуанчжоу Москва» → получает расчёт
- Отправляет голосовое на русском или китайском → бот транскрибирует и обрабатывает
- Спрашивает «где мой груз?» → получает текущий статус
- Получает автоуведомления при смене статуса фуры

### Голосовые сообщения (STT)

**Flow:**
1. Клиент/менеджер отправляет голосовое в Telegram
2. OpenClaw перехватывает аудио
3. Whisper (локальный, на GPU) транскрибирует: ~3–5 сек
4. Транскрипт передаётся дальше как обычный текст → skills обрабатывают

**Языки:** русский, китайский — оба поддерживаются Whisper из коробки.

**Примеры:**
- Менеджер 🎤 «фура двадцать пять прошла границу» → status_skill → 50 уведомлений
- Клиент 🎤 «сколько будет стоить восемьсот штук кроссовок вот с этой ссылки» + ссылка → calc_skill → расчёт
- Китайский клиент 🎤 «八百双运动鞋从广州到莫斯科多少钱» → транскрипт на китайском → Qwen понимает → расчёт

**Почему это важно:** менеджеры карго живут в мессенджерах и привыкли общаться голосом. Печатать «фура 025 статус border» — неестественно. Наговорить голосовое — естественно. Это убирает последний барьер к использованию.

**Себестоимость:** 0 руб (Whisper локальный, GPU уже работает).

## 4. Skills (что разрабатываем)

### 4.1 parser_skill — парсинг 1688

**Триггер:** получена ссылка на 1688.com или detail.1688.com

**Действие:**
1. Playwright headless открывает страницу
2. Извлекает: название товара, цена за штуку (¥), вес (кг), габариты (Д×Ш×В см), фото URL
3. Возвращает структурированные данные

**Edge cases:**
- Страница не загрузилась → «Не могу открыть страницу. Проверьте ссылку или попробуйте позже.»
- Нет веса на странице → возвращает null, calc_skill попросит указать вручную
- Антибот защита 1688 → retry с другим user-agent, при неудаче → просим ввести данные вручную

**Данные на выходе:**
```json
{
  "title": "女款运动鞋",
  "price_cny": {"min": 45.0, "max": 60.0, "variants": [{"name": "36-39", "price": 45.0}]},
  "weight_kg": 0.3,
  "dimensions": {"l": 30, "w": 20, "h": 12},
  "image_url": "https://...",
  "offer_id": "790251400429",
  "source": "scrapling_llm"
}
```

**Адаптер:** `adapt_parser_output()` в calculator.py преобразует `price_cny.min` → `price_per_piece_cny: float` для калькулятора.

### 4.2 calc_skill — расчёт стоимости

**Триггер:** клиент отправил ссылку + количество, или текстовый запрос с параметрами

**Входные сценарии:**

Сценарий А (ссылка):
```
Клиент: [ссылка на 1688] 800 штук
```

Сценарий Б (текст):
```
Клиент: 500 кг одежда из Гуанчжоу в Москву
```

**Действие:**

Сценарий А:
1. Вызывает parser_skill → получает данные товара
2. Рассчитывает: кол-во × вес = общий вес, объёмный вес
3. Вызывает Haiku: классификация товара → категория, хрупкость, доп. услуги
4. Загружает ставки компании из rates/{company_id}.json
5. Считает стоимость по каждому виду транспорта
6. Формирует ответ

Сценарий Б:
1. Вызывает Haiku: парсинг параметров из текста (вес, товар, откуда, куда)
2. Шаги 4–6 из сценария А

**Расчёт:**
```
закупка = цена_за_шт × количество
общий_вес = вес_за_шт × количество
общий_объём = (Д × Ш × В × количество) / 1 000 000  (м³)
плотность = общий_вес / общий_объём  (кг/м³)

для каждого транспорта:
  находим нужный диапазон плотности в density_rates
  если rate_per_kg → стоимость = общий_вес × rate_per_kg
  если rate_per_m3 → стоимость = общий_объём × rate_per_m3

категория товара (Haiku):
  если electronics → × category_surcharges.electronics
  если fragile → + crating_pct (40%)

доп_услуги:
  обрешётка = +40% (если хрупкое)
  паллетирование = +16% (если тяжёлое оборудование)
  страховка = 3% от стоимости товара (если дорогое)
```

**Формат ответа:**
```
Кроссовки женские
800 шт | 240 кг | 1.2 м³ | Плотность: 200 кг/м³ | ¥45/шт

Закупка: ¥36 000 (~$4 960)

Доставка (ваши ставки, плотность 200):
  Авто: $2.80/кг → $672 | 18–25 дн
  ЖД:   $2.30/кг → $552 | 25–35 дн
  Авиа:  $6.50/кг → $1 560 | 5–7 дн

Доп. услуги: не требуются

Итого (авто): ~$5 632
За штуку: ~$7.04
```

**Edge cases:**
- Нет веса (ни с 1688, ни из текста) → «Укажите вес груза, чтобы я посчитал»
- Неизвестная категория → считает как «general cargo» без доп. услуг
- Нет ставок на запрошенный маршрут → «У компании нет ставок на этот маршрут. Обратитесь к менеджеру.»

### 4.3 status_skill — статусы фур

**Триггеры менеджера (естественный язык):**
- «фура 025 выехала из Гуанчжоу» → статус: departed
- «025 на границе» → статус: border
- «фура 025 прошла таможню» → статус: customs
- «025 в Москве» → статус: moscow
- «фура 025 — выдача» → статус: delivered
- «покажи все фуры» → список активных фур со статусами

**Статусы и шаблоны уведомлений:**

| Код | Статус | Сообщение клиенту |
|---|---|---|
| warehouse | На складе в Китае | «Ваш груз принят на склад в Китае. Ожидайте отправки.» |
| packed | Упакован | «Ваш груз упакован и готовится к отправке.» |
| departed | Отправлен | «Ваш груз отправлен! Маршрут: {маршрут}. Ориентировочно: {срок}.» |
| border | На границе | «Ваш груз на погранпереходе. Ожидание прохождения.» |
| customs | Таможня пройдена | «Ваш груз прошёл таможню, в пути до {город}. Ориентировочно: {срок}.» |
| moscow | В городе назначения | «Ваш груз прибыл в {город}! Свяжитесь для получения.» |
| delivered | Выдан | «Ваш груз выдан. Спасибо за доверие!» |

**Триггер клиента:**
- «где мой груз?» / «статус» / «когда приедет» → бот ищет клиента в trucks.db → отвечает текущим статусом

## 5. Мультитенант: каждая компания = свой бот

### Архитектура

Каждая карго-компания получает один Telegram-бот. Менеджер и клиенты компании общаются через него же. Роль определяется по Telegram ID отправителя.

```
@FastCargo_bot  ─┐
@SpeedCargo_bot ─┼→ cargo-agent → по токену определяет компанию → проверяет ID → роль
@ThirdCargo_bot ─┘
```

- Telegram ID отправителя == `manager_telegram_id` → менеджер (полный доступ)
- Все остальные ID → клиент (только расчёты и «где мой груз»)

### База данных (единая SQLite → PostgreSQL при росте)

```sql
CREATE TABLE companies (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    bot_token TEXT UNIQUE NOT NULL,
    manager_telegram_id TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    subscription_tier TEXT DEFAULT 'start'
);

CREATE TABLE company_rates (
    company_id TEXT PRIMARY KEY REFERENCES companies(id),
    rates_json TEXT NOT NULL
);

CREATE TABLE trucks (
    id TEXT NOT NULL,
    company_id TEXT NOT NULL REFERENCES companies(id),
    route TEXT,
    status TEXT DEFAULT 'warehouse',
    status_updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id, company_id)
);

CREATE TABLE truck_clients (
    truck_id TEXT NOT NULL,
    company_id TEXT NOT NULL,
    client_telegram_id TEXT NOT NULL,
    client_name TEXT,
    cargo_description TEXT,
    FOREIGN KEY (truck_id, company_id) REFERENCES trucks(id, company_id)
);

CREATE TABLE client_company_map (
    client_telegram_id TEXT NOT NULL,
    company_id TEXT NOT NULL REFERENCES companies(id),
    linked_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (client_telegram_id, company_id)
);
```

### Как работает привязка клиента

1. Менеджер создаёт бот через @BotFather → токен
2. Менеджер пишет в этот же бот → визард онбординга → компания регистрируется
3. Менеджер проходит визард ставок → rates сохраняются
4. Бот компании подключается к OpenClaw
5. Менеджер даёт ссылку на бот своим клиентам
6. Клиент пишет в @FastCargo_bot → автоматически привязывается к FastCargo
7. Все расчёты/статусы — по данным FastCargo

### OpenClaw конфигурация

Каждый новый бот-токен добавляется как новый Telegram account в openclaw.json, привязанный к cargo-agent. Agent определяет компанию по токену бота → загружает ставки → проверяет Telegram ID отправителя против `manager_telegram_id` → выдаёт соответствующий уровень доступа.

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| 1688 блокирует парсинг | Средняя | Высокое | Ротация user-agent, прокси, fallback на ручной ввод |
| Домашний сервер упал | Низкая | Высокое | UPS, мониторинг, план миграции на VPS при росте |
| Qwen не понимает запрос | Средняя | Среднее | Fallback на Haiku для сложных случаев |
| Карго-компании не хотят платить | Средняя | Высокое | Бесплатный пилот → доказать ценность на цифрах |
| Ставки компании устарели | Высокая | Среднее | Напоминание менеджеру при обновлении курсов |
